# S2 Approach

## Scope and Authority

This approach maps the six provider-neutral paper-search actions to the Semantic Scholar Academic Graph and Recommendations APIs. It is self-contained operational guidance based on the official [Academic Graph API documentation](https://api.semanticscholar.org/api-docs/graph), [Recommendations API documentation](https://api.semanticscholar.org/api-docs/recommendations), and [API tutorial](https://www.semanticscholar.org/product/api/tutorial), checked on 2026-07-23.

The initial approach excludes author search, snippet search, the Datasets API, bulk corpus export, and unrelated endpoint recipes. S2 names and response wrappers remain in this reference and never become required fields in the provider-neutral observation schema.

## Action-to-Operation Map

| Paper-Search Action | S2 Operation Intent | Response Used |
| --- | --- | --- |
| `resolve-paper` | Paper detail for a stable id; title match or tightly bounded relevance search for ambiguous title input; paper batch only for an already bounded id set | One paper object, bounded match candidates, or batch positions |
| `search-papers` | Relevance search for ranked results; bulk search for broader token-paginated retrieval; title match for exact-title resolution | `data` candidates plus offset, next, total, or token posture |
| `find-citing-papers` | Paper citations | `data[].citingPaper` and the target as cited endpoint |
| `explore-cited-papers` | Paper references | `data[].citedPaper` and the target as citing endpoint |
| `trace-citation-neighborhood` | Repeated bounded citation and reference operations plus optional batch metadata for returned ids | Per-hop citing and cited wrappers mapped into intrinsic edges |
| `find-related-papers` | Single-seed recommendations or multi-seed positive and negative recommendations | `recommendedPapers` as non-citation candidates |

Use only the operations needed for the selected action. Do not fetch authors, embedded citation lists, or broad abstracts when normalized identity and bibliographic fields suffice.

## Supported Paper Identifiers

Paper detail and graph operations accept a raw S2 paper id and qualified identifiers such as `CorpusId:`, `DOI:`, `ARXIV:`, `MAG:`, `ACL:`, `PMID:`, `PMCID:`, and supported publication URLs. Preserve the requested form and the resolved `paperId`. URL-encode all path identifiers; never concatenate untrusted text into a path.

For title input, treat title match as identity assistance rather than authority. Several plausible matches remain an ambiguity blocker. Paper batch accepts at most 500 ids and can return null positions; preserve input order, null entries, and unresolved ids.

## Minimal Field Projection

Start with `paperId,corpusId,externalIds,url,title,authors,venue,publicationDate,year`. Add only action-required fields. Use the `fields` parameter and dot notation where the selected operation requires nested fields. Do not request citation contexts, intents, influence labels, abstracts, embeddings, or open-access payloads unless the caller's declared purpose needs them and the evidence boundary remains explicit.

Map:

- `paperId` to a provider-qualified identity and candidate provider id;
- `externalIds.DOI` and `externalIds.ArXiv` to normalized external identifiers;
- `url` to a provider locator;
- `authors[].name`, `venue`, `publicationDate`, and `year` to optional normalized bibliographic fields;
- missing or null values to `missing_fields`, `null_records`, or `unresolved_records`.

## Response Shapes and Direction

Paper detail returns one paper object. Paper batch returns an array aligned to submitted ids and may contain null entries. Relevance, bulk, and title searches return candidate data plus their operation's pagination posture.

The citations operation returns edge wrappers containing `citingPaper`; normalize each edge as `citingPaper -> target` with route `forward`. The references operation returns wrappers containing `citedPaper`; normalize each edge as `target -> citedPaper` with route `backward`. Preserve null wrappers and null nested papers rather than treating them as negative findings.

Recommendations return `recommendedPapers`. They are adjacent-paper observations, not citation edges. The single-seed operation and the positive or negative multi-seed operation have a bounded `limit` of at most 500 and no documented continuation token; reaching the requested limit is not evidence of exhaustive adjacency.

## Pagination and Filtering

Relevance search and citation or reference lists use offset pagination and return continuation posture such as `next`; record every requested and returned offset. Bulk search uses the returned opaque token. Copy it exactly and stop at provider exhaustion or the declared result or page bound.

Use `publicationDateOrYear` or `year` only on operations that document the filter, including paper search variants and citations where available. Paper detail, references, and recommendations do not provide an equivalent general date filter. For those operations, retrieve only a declared bounded set, apply the explicit inclusive date or year range locally, and record `filtering.location` as `local` or `mixed`. A null publication date can coexist with a year; preserve the provider values and the applied rule.

## External Tool and HTTPS Execution

Use a provider-native or general-purpose CLI when available under the Literature Provider Binding. A permitted direct-HTTPS fallback uses the documented service root, one action-specific path, safe path encoding, `--get` with `--data-urlencode` for query values, a minimal `fields` projection, a finite connection and total timeout, and a file-backed JSON response when raw retention is approved.

An anonymous query shape may be constructed without embedding user text:

```bash
curl --fail-with-body --silent --show-error --get \
  --connect-timeout 10 --max-time 60 \
  --data-urlencode "query=QUERY" \
  --data-urlencode "fields=paperId,externalIds,url,title,authors,venue,publicationDate,year" \
  --data-urlencode "limit=RESULT_LIMIT" \
  "https://api.semanticscholar.org/graph/v1/paper/search"
```

Replace placeholders through tool arguments, not shell interpolation of untrusted text. When authentication is approved, inject the `x-api-key` header through the tool's secret-safe header mechanism without rendering the value or a credential-bearing command. Never put the key in a URL.

## Credentials, Throttling, and Retry

Resolve an S2 key only from an approved credential binding or a non-empty `S2_API_KEY` process environment. Do not scan `.env` files or other arbitrary files. The official API overview recommends keys and documents that authenticated and anonymous access have different throttling posture; record the observed access method and limitations.

Treat 429, timeout, connection reset, and transient 5xx responses as retryable within the declared retry and elapsed-time bounds. Honor `Retry-After` when supplied. Treat 400, 401, 403, 404, ambiguity, and policy denial as terminal until input or authorization changes. Keep successful pages and return a partial observation after a later failure.

Keys, authorization headers, signed locators, and secret-bearing requests or responses must not appear in URLs, commands shown to users, stdout or stderr captures, chat, normalized observations, Artifacts, raw attachments, or Provenance Records.

## Normalization and Recording

Map every S2 response into `isomer-literature-provider-observation.v1` according to `references/result-contract.md` before recording. Include provider `semantic-scholar`, approach `s2`, access method, request intent, requested and applied bounds, papers, citation edges, pagination, filtering location, completeness, missing and null records, partial failures, limitations, and Provenance refs.

Do not invoke an S2 search, detail, recommendation, citation, or reference operation through `isomer-cli`. Do not ask `isomer-cli` to interpret S2 wrappers. After agent-side normalization, use `isomer-cli ext research literature record --payload-file` only for local validation, canonical observation recording, projection refresh posture, and later local queries.

Raw S2 responses are optional redacted attachments. Retain one only when policy permits and diagnosis or reproducibility needs it. Store media type, SHA-256 checksum, redaction posture, and Provenance refs; normalized indexing never reads the attachment.

# Core Welcome Help

## Workflow

1. Explain that `$isomer-op-welcome` teaches typical use cases and remains read-only, while `$isomer-op-entrypoint` routes and proceeds with concrete core work.
2. List `show-options`, `show-extensions`, `choose-path`, `show-command-map`, `next-step`, the four start-path commands, and `help`.
3. Present several high-value newcomer examples from [show-options.md](show-options.md).
4. Offer [show-command-map.md](show-command-map.md) only when the user wants the exhaustive execution inventory.
5. End with one recommended public invocation or a request for the smallest missing goal.

If the request does not map cleanly to these steps, use the native planning tool to build concise help from the public welcome commands, use-case patterns, output contract, and read-only boundary.

## Quick Start

Use `$isomer-op-welcome` to compare core platform paths. Use `$isomer-ext-deepsci-welcome` or `$isomer-ext-kaoju-welcome` to learn an optional research paradigm. Once the task is concrete, send it to the matching entrypoint with `$<entrypoint> use <command> to <task>` or as a task-only request.

Examples are guidance, not authorization. A recommended command may still encounter prerequisites, mutation boundaries, network or publication Gates, or a need for user approval.

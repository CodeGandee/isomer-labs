# Start a Kaoju Survey

## Workflow

1. Interpret the goal as evidence-led survey work over literature, code, datasets, models, or first-hand trials, independent of manual or formal Agent Team topology.
2. Recommend `$isomer-ext-kaoju-welcome` when the user needs typical use cases, evidence-stage distinctions, readiness guidance, or command learning.
3. When the task is concrete, match it to a public Kaoju command and recommend `$isomer-ext-kaoju-entrypoint use <command> to <task>` with supplied context.
4. Identify any scope clarification, source selection, template choice, execution approval, network exposure, or other Gate before the example.
5. Explain missing Project, Topic Workspace, survey, repository, dataset, environment, extension, or Kaoju readiness and recommend the public core or Kaoju entrypoint that owns recovery.

If the task does not map cleanly to these steps, use the native planning tool to compare framing, discovery, intake, examination, comparison, bounded trial, synthesis, paper, and wiki patterns, then recommend the extension welcome or one concrete command.

## Mutation Boundary

This routine does not install Kaoju, acquire sources, prepare an environment, run code, write a paper, export a wiki, or approve a later Gate. The example command starts a governed procedure; it does not preapprove later mutations or external actions.

# Choose a Core Path

## Workflow

1. Decide whether the user wants orientation or has already supplied an actionable task.
2. Match the goal to one row in [show-options.md](show-options.md), preserving all supplied Project, Topic, actor, agent, extension, Toolbox, environment, and research context.
3. Separate execution topology from research paradigm before recommending manual, formal Agent Team, DeepSci, or Kaoju paths.
4. Recommend one public welcome for further learning or one public entrypoint invocation for execution.
5. State the expected action, mutation posture, missing prerequisite or decision, and next step without invoking the route.

If the user's task does not map cleanly to these steps, use the native planning tool to compare the public patterns and ask for the smallest missing distinction rather than guessing or exposing protected routes.

## Selection Rules

Choose `$isomer-op-welcome` when the user still needs core workflow comparison. Choose `$isomer-ext-deepsci-welcome` or `$isomer-ext-kaoju-welcome` when the research paradigm is known but its typical uses or commands are not.

Choose `$isomer-op-entrypoint use <command> to <task>` when the user has a concrete Project, Topic, GUI, identity, extension, Toolbox, formal-team, or environment task. Choose the corresponding extension entrypoint when a DeepSci or Kaoju task and its relevant context are already concrete.

Representative phrases guide semantic routing but are never required literal keywords. A selected example does not preapprove mutation or a later Gate.

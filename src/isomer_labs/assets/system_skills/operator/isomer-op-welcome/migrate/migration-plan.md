# Core Welcome Migration Plan

## Scope

Migrate the protected source at `operator/isomer-op-entrypoint/subskills/isomer-op-welcome` to the independent public target `operator/isomer-op-welcome`. This is a source-to-target structural migration with an Imsight-native runtime rewrite.

## Source Behavior to Preserve

Preserve the read-only menu, topology-versus-paradigm distinction, extension evidence ladder, Project-context recommendation, four research-start routines, owner handoff, essential and complete outputs, and the rule that welcome never executes its recommendation.

## Target Runtime Shape

The target owns `SKILL.md`, `agents/openai.yaml`, and one-level `references/` pages. It uses peer subcommands because every routine shares the welcome resource boundary. It owns no `subskills/`, scripts, execution procedure, callback insertion points, or protected capabilities. The former `show-skill-map` resource becomes the newcomer-facing `show-command-map` resource, while the entrypoint retains a compatibility command that delegates to welcome.

## Term Adaptations

Replace protected-member and parent-scoped welcome language with the public `$isomer-op-welcome` identity. Describe execution only through the public core, DeepSci, and Kaoju entrypoints. Add `welcome-seen` beside `entrypoint-seen` and replace receipt v4 integrity language with the current multi-public receipt contract.

## Tool and Harness Adaptations

Retain only read-only `isomer-cli` inspection commands. Installation, upgrade, Project mutation, Topic work, research execution, and Toolbox work become recommended public entrypoint invocations.

## Artifact and Storage Adaptations

Welcome creates no durable artifact or storage binding. The audit snapshot and analysis remain passive under `org/`; this plan remains passive under `migrate/`.

## External Skill Route Adaptations

Core orientation uses `$isomer-op-welcome`; extension orientation uses `$isomer-ext-deepsci-welcome` or `$isomer-ext-kaoju-welcome`; concrete work uses the corresponding execution entrypoint. No active welcome page advertises a protected logical id or parent-owned designator.

## Step Support Mapping

The target entrypoint keeps route selection and output rules concise. Typical-use-case fields live in `references/show-options.md`; exhaustive command coverage lives in `references/show-command-map.md`; extension evidence and read-only context boundaries remain in dedicated references.

## Semantic Match Checks

The rewrite preserves all ten public welcome routines, read-only context inspection, owner handoff, topology and paradigm separation, and output depth. It intentionally changes the invocation from a protected subskill to a public sibling, adds two extension welcomes, replaces direct protected-route teaching with public commands, and adds newcomer use-case fields required by the approved OpenSpec change. No unresolved placeholders remain.

# Closure Limits

## Workflow

When deciding whether GPU analytical-modeling work can close, execute the following steps in order.

1. **Identify central contribution claims**. Do not treat formatting, packaging, or prose polish as enough for closure.
2. **Check central evidence**. Runtime, counter-trend, saturated-component, and blocking-path claims each need matching evidence or an explicit limitation.
3. **Choose closure posture**. Publish, park, defer, limit, route back, or block according to the weakest central claim.
4. **Write reopen conditions**. If parked or deferred, name what evidence or implementation would change the decision.
5. **Preserve resume context**. Tell the next agent what to read first and which failed or partial evidence must not be repeated blindly.

If the user's task does not map cleanly to these steps, use your native planning tool to build a closure plan from the central claims, evidence gaps, closure postures, and resume context, then execute the plan. If the evidence does not support a central claim, do not close as publishable.

## Closure Postures

- Publish: central claims supported for stated scope.
- Park: plausible work with missing central evidence or venue decision.
- Defer: noncentral claim intentionally moved out of scope.
- Limit: claim retained only in a narrower scope.
- Route back: next action is experiment, analysis, review, or decision.
- Block: progress needs user input, access, repair, or external state.

## Common Mistakes

- Treating a polished PDF as closure evidence.
- Failing to name the exact missing evidence for a parked result.
- Letting "future work" replace a clear limitation.
